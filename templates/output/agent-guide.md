## Before You Start Coding

**Confirm the environment.** The plan tells you the technology stack but not the exact setup. Before writing a single line of code, ask the developer:

- How should the environment be structured? (Docker Compose layout, specific services, port mappings)
- Which versions of each tool? (PHP, Node, database — specific versions, not ranges)
- What's already in place vs. what needs to be created?
- Are there any local development preferences or constraints?

Do not assume. A wrong Docker setup takes longer to undo than it takes to ask. Treat environment confirmation as the implicit first task — no coding until it's resolved.

**Respect Kanban dependencies.** If a task depends on another, do not start it until the dependency is resolved. The Kanban is your execution order, not a suggestion.

## How to Act on This Plan

1. **Read the documents in order above.** The executive summary orients you; the overview gives depth; the requirements tell you what to build; the Kanban tells you in what order.

2. **Respect the guardrails.** If the plan defines boundaries (scale, budget, risk, project type), enforce them. Do not suggest work outside scope without flagging it.

3. **Follow the confidence levels.** Areas marked `High` are well-understood — treat them as settled. Areas marked `Medium` or `Low` have assumptions that may need validation. Flag these before acting on them.

4. **Keep the plan alive.** This is not a static reference — update it as you work:

   - **Kanban:** Add new tasks as you discover them. Mark tasks complete when done. Update dependencies if the order shifts.
   - **Design decisions:** When you make a non-obvious choice (database schema, API structure, library selection), document it in the relevant state file or a new decision record. Include *why* — the rationale matters more than the outcome.
   - **Risks:** Add new risks as they emerge. Update likelihood or impact if they change.
   - **Deprecations:** If an assumption in the plan proves wrong, log it — don't silently overwrite. What changed, why, and what replaces it.
   - **Confidence:** As unknowns become known, update the confidence assessment.

   A plan that doesn't reflect reality is worse than no plan at all.

5. **Check open questions.** Any unresolved items in the Open Questions section should be addressed before they become blockers.

6. **Use the Kanban as a starting point.** Tasks are prioritized (P0–P3) with effort estimates. Dependencies between tasks are noted — respect the dependency order.
